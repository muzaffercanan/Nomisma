Fintech domain'inde kredi yönetimi yapan bir REST API. Müşteri kaydından başlayıp kredi skoru kontrolü, taksit planı üretimi ve ödeme gateway entegrasyonuna uzanan gerçek dünya senaryolarını kapsıyor. Sade CRUD değil — iş kuralı, rol tabanlı erişim ve atomik işlem gerektiren bir domain.

Tasarım Kararı — 09

Static Mapper (AutoMapper Değil)

AutoMapper yaygın bir kütüphane ama bu projede bilinçli olarak static mapper metodlar tercih edildi. Bu küçük görünen karar büyük farklar yaratıyor.

❌ AutoMapper ile (değil)
// Mapping profile ayrı dosyada
// Runtime'da çözülüyor
// Yanlış mapping → null döner
// Debug stack trace belirsiz
// Ek NuGet bağımlılığı
// Reflection overhead

var dto = _mapper.Map<CustomerResponseDto>(customer);
✅ Static Mapper ile (tercih edilen)
// Kod tamamen görünür
// Compile-time kontrol
// Yanlış mapping → derleme hatası
// Debug'da tam stack trace
// Sıfır bağımlılık
// Sıfır reflection

var dto = CustomerMapper.ToDto(customer);
CustomerMapper Implementasyonu Explicit
Application/Customers/CustomerMapper.csC#

Her property assignment açıkça görünüyor. Hangi alan nereye gittiği kod okuyarak anlaşılıyor — ayrı bir mapping konfigürasyonu aramak gerekmiyor.

Neden Makul Bir Karar? Trade-off
  • Derleme zamanı güvenliği: Property ismi değişirse mapper'da derleme hatası alırsın — AutoMapper'da runtime null.
  • Debug edilebilirlik: Stack trace'de tam olarak hangi satırda ne çevrildiği görünüyor.
  • Sıfır bağımlılık: AutoMapper bir NuGet paketi — her versiyon güncellemesinde breaking change riski var.
  • Performans: Reflection yok, doğrudan property atama.
  • Sınırlı entity sayısı: 4 entity için tek tek yazmak makul. 40 entity olsaydı karar değişebilirdi.
